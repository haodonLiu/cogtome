# ELT 范式：让 LLM 做它擅长的事

> 核心思路：让 LLM 做它擅长的事（语义判断与因果推理），让代码做它擅长的事（精确格式化与去重），并且让 LLM 尽早接触原始数据，而不是被前面的规则环节"嚼碎"后的二手信息。

---

## 一、根本问题诊断：当前架构在"逆着 LLM 的能力特性"工作

你当前的流水线是典型的人类预处理思维：

```
原始文本 → [人写规则切分] → [人写规则统计] → [人写模板] → LLM 填个空
```

这相当于把 LLM 当成高级模板引擎在用。但 LLM 真正的优势是端到端的语义理解和跨片段的关联推理。当前架构恰恰在剥夺它的这些能力：

- 信息逐层失真：`memory-indexer` 读原始文本 → `task-extractor` 用正则砍成碎片 → `pattern-finder` 只拿到词频 → `report-writer` 拿到的是已经被压缩成标签的信号。LLM 在最后一步就算想分析，也无法还原原始语境了。
- 规则承担了不该承担的语义判断：正则匹配 emoji 假设"用户会规范使用标记"，词频统计假设"出现即重要"。这两个假设在真实人类行为中都不成立。

所以我的第一性原理是：把架构从 ETL 翻转成 ELT。

| 范式 | 流程 | 适用场景 |
|------|------|----------|
| ETL（当前） | Extract（正则切任务）→ Transform（词频统计）→ Load（给 LLM 填空） | 传统数据仓库，schema 严格 |
| ELT（我提议的） | Extract（读原始数据）→ Load（把原始上下文喂给 LLM）→ Transform（让 LLM 做语义判断） | LLM 时代，语义理解优先 |

## 二、每个设计决策的具体 rationale

### 1. 为什么 task-extractor 要从"纯正则"改为"正则粗筛 + LLM 精筛"？

不是因为正则不够用，而是因为"任务"本身是一个语义概念，不是语法概念。

- 正则的隐含假设：用户用 `✅` 标记完成的任务，用 `🔄` 标记进行中的任务。但人类记日记时 emoji 是情绪修饰，不是结构化标记。你可能在吐槽"🔄 又是改配置"（其实是抱怨，不是任务），也可能在写"终于把 Phase 1 搞定了"（没 emoji，但确实是 completed）。
- LLM 能识别行动意图（agency）：它能区分"今天很累"（状态）和"今天重构了 auth 模块"（行动），这种区分需要上下文理解，正则做不到。
- 为什么保留正则作为粗筛：成本。LLM token 不便宜，先用正则筛掉明显无关的段落（纯情绪、纯天气记录），减少无效调用。这是工程权衡，不是能力问题。

### 2. 为什么 pattern-finder 要从"词频"改为"语义聚类 + LLM 诊断"？

因为"高频词"和"行为模式"是两个完全不同的东西。

- 词频统计的陷阱：假设"skill"出现 3 次就是"skill 模式"。但可能你只是在讨论 skill 架构（"skill 应该怎么设计"），而不是在反复执行 skill 相关工作。词频无法区分"提及"和"实践"。
- 真正的模式是什么？是重复出现的困境结构或重复使用的解决路径。比如：
  - "周三深夜改 Docker 配置"
  - "周五深夜改 Docker 配置"
  → 模式不是"Docker"（词频会报这个），而是"深夜处理 infra 问题"（时间 + 任务类型 + 认知负载的交叉模式）。
- 这需要 LLM 做跨维度关联：把时间、任务类型、情绪、trace 异常放在一起看。词频统计是单维度的，注定看不到这种结构。

### 3. 为什么单独抽出 reflection-engine，把 report-writer 降级为"纯格式化"？

这是"关注点分离"（Separation of Concerns）在 Prompt 工程里的应用。

当前 `report-writer` 既要做因果推理（"为什么总在深夜工作？"），又要做模板填充（"把结果塞进 markdown"）。这两个职责的迭代频率和失败模式完全不同：

| 职责 | 迭代频率 | 失败表现 | 调试难度 |
|------|----------|----------|----------|
| 因果推理 | 高（每次想改进分析深度） | 得出错误结论 | 难（要追踪思维链） |
| 模板填充 | 低（格式定好后很少改） | 输出格式错乱 | 易（看 JSON 是否解析成功） |

混在一起的结果是：你想调分析深度，却破坏了输出格式；你想修格式，却不小心删掉了推理约束。

抽离后的好处：
- `reflection-engine` 可以专注做"审计式分析"：追问根因（5 Whys）、交叉验证（任务说 ✅ 但 trace 报错）、预测风险。它的 prompt 可以很长很复杂，因为不需要担心格式。
- `report-writer` 变成"JSON → Markdown 的确定性转换"，甚至可以完全不用 LLM，用 Jinja 模板即可。

### 4. 为什么去重要从"文本前缀"改为"语义相似度"？

因为人类对同一任务的表述是"意合"而非"形合"的。

- `text[:60]` 去重的假设：同一任务的不同记录总是开头相似。但实际情况：
  - "Phase 1 实现"
  - "Phase 1 实现：加了 foreach 支持"
  - "foreach 支持搞完了，Phase 1 收尾"
  这三条前缀完全不同，但指向同一任务。前缀匹配会把它们当成三个任务，导致报告里"完成 15 个任务"的虚假繁荣。
- 为什么用 `SequenceMatcher`（编辑距离）而不是向量相似度：在这个场景下，编辑距离足够好，且不需要引入 embedding 模型或向量数据库。这是"够用即可"的工程保守主义——如果后续发现编辑距离误判太多（比如"修 bug"和"写 bug"相似度高但语义相反），再升级到语义向量也不迟。

### 5. 为什么 trace 要强制作为"证据链"融入 reflection？

因为当前系统存在"自我报告偏差"（Self-report Bias）。

- 任务记录是用户的主观叙事：你标记 `✅` 可能是因为"觉得思路通了"，但实际上 trace 里还有 3 次重试和 1 个未处理的异常。
- 如果 reflection 只看任务记录，它会基于被美化过的数据得出错误结论。比如报告说"本周完成率 90%"，但 trace 显示 40% 的 completed 任务后续还有错误日志。
- 交叉验证的设计：强制要求每个模式判断都关联 trace 证据，这借鉴了审计学中的"corroborating evidence"原则——单一来源的信息不可信，需要独立证据源印证。

## 三、优先级排序的底层逻辑

为什么我建议 P0 先改 reflection prompt，而不是先改上游 Extractor？

这是"防御性设计"（Defensive Design） 的思路：

| 策略 | 比喻 | 风险 |
|------|------|------|
| 先修上游 | 先净化水源，再修水龙头 | 上游涉及多个模块，改动周期长；期间用户一直喝不到干净水 |
| 先修下游 | 先给水龙头装过滤器，再慢慢修水管 | 即使上游数据有噪音，下游的强 LLM 能通过上下文理解部分纠错；用户立即感受到质量提升 |

reflection-engine 作为下游，可以通过强 prompt 约束来容忍上游的一定噪音。比如：

- 即使 extractor 把"Phase 1 实现"和"Phase 1 实现：foreach"当成两个任务，reflection-engine 的 prompt 里明确要求"识别同一任务的不同表述并合并"，LLM 有能力在分析阶段把它们聚类。
- 即使 pattern-finder 报了一个虚假的"skill 模式"，reflection-engine 可以通过追问"这些 mention 是在讨论还是在实践？"来过滤掉噪音。

**先让下游能"容错"，再逐步把上游的噪音源掐掉。** 这样你每改一个模块，用户都能感知到正向变化，而不是陷入"大重构半年看不到成果"的泥潭。

## 四、总结：这个架构的元原则

如果把这些具体决策抽象成一个可复用的设计框架，它是这样的：

```
1. LLM 介入点 = "需要判断的地方"，而非"需要格式化的地方"
2. 数据流向 = "原始数据尽量晚被压缩"，让 LLM 在信息最丰富的阶段做决策
3. 模块边界 = "按思维职责切分，而非按数据形态切分"
4. 容错策略 = "下游先强壮，上游逐步净化"
5. 工程权衡 = "能用规则精确做的事，不麻烦 LLM；需要语义理解的事，不硬写规则"
```

这套思路不只适用于 Midnight Reflection，也适用于任何"让 LLM 分析人类行为记录"的系统。核心就是：**别让人类写的规则去猜语义，别让 LLM 去填被规则榨干了的空。**
