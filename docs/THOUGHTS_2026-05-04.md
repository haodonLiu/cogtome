# COGTOME 思考碎片整理

**日期：** 2026-05-04
**参与：** Easton + Hermes

---

## 一、COGTOME 的核心定位

**COGTOME = LLM 的工作流 IDE + 仪表盘**

### 核心价值
- **Atomic（原子化）**：每个 Unit 独立可执行，边界清晰
- **Traceable（可追踪）**：全程记录输入/输出/耗时，出了 bug 能倒查

### 定位对比

| 传统方式 | COGTOME 方式 |
|---------|-------------|
| "我觉得这部分慢" | trace 数据告诉你哪个节点多少毫秒 |
| "应该优化这里" | 优化后 trace 对比，数字说话 |
| "跑了，没报错" | 每一步的输入/输出/耗时都有记录 |
| LLM 自己规划工作流 | LLM 规划 + COGTOME 执行 + trace 验证 |

---

## 二、COGTOME 不适合什么

**"自优化"在 COGTOME 内部走不通。**

尝试过的路径：
- 通过改 `workflow.json` 合并 Unit（Plan B）→ COGTOME 不支持 motif 内联脚本执行
- 通过改 workflow 加缓存（Plan C）→ 同上，motif 是 JSON graph，不是可执行脚本

**根本原因：** COGTOME 的 Motif 是 JSON workflow graph，节点只能是 Unit 或其他 Motif 的引用，没有"内联执行 Python 逻辑"的机制。

---

## 三、正确的协作模式

```
用户说："帮我整理这周的工作"
     ↓
LLM 分解任务：
  - 需要读 memory → memory-indexer
  - 需要提取任务 → task-extractor
  - 需要分析模式 → pattern-finder
  - 需要生成报告 → report-writer
     ↓
LLM 编排成 workflow → COGTOME 执行（带 trace）
     ↓
trace 回来 → LLM 分析结果 → 发现问题 → 修改 workflow → 重新执行
```

**角色分工：**
- **LLM** = 规划工具（理解需求、设计 workflow、分析 trace、迭代改进）
- **COGTOME** = 带仪器的执行引擎（原子化执行、全程 trace、可审计可回放）

**不是：** LLM 自己优化自己
**而是：** LLM 借助 COGTOME 的 trace 数据做有依据的优化决策

---

## 四、优化的工作流程

```
COGTOME 执行 + trace → 我（LLM）读 trace → 定位问题 + 发现优化点 → 用户决定 → 我执行修改
```

"优化不一定非得有我们实现，我们的目的只是帮用户高效的编排工作流，并帮他们定位问题，最后帮他们找到可以优化的地方。"

## 五、WebUI 与 Trace Dashboard 的现状

### WebUI（`webui/`）
- React + TypeScript + ReactFlow
- 功能：DAG 可视化编辑、节点拖拽、保存/读取 workflow
- 路由：StructureList / MotifList / UnitList / 各 Editor

### Trace Dashboard（`trace_dashboard.rs`）
- 独立的 Rust HTTP 服务
- 功能：统计页面（成功率、平均耗时）、节点耗时列表
- API：`/api/traces`、`/api/traces/stats`、`:trace_id`

### 差距分析

**已有的：**
- DAG 可视化编辑 ✅
- Trace 统计数字 ✅

**缺失的（需要在 DAG 上展示）：**
- ❌ 实时执行状态（节点执行时高亮）
- ❌ 每步的 avg_ms、成功率
- ❌ Trace 数据与 DAG 节点关联展示
- ❌ 仪表盘监控视图

## 六、产品方向（目标态）

```
┌─────────────────────────────────────────────────────┐
│  COGTOME WebUI                                        │
├───────────────────────┬─────────────────────────────┤
│                       │                              │
│   Workflow 可视化      │      监控仪表盘              │
│                       │                              │
│   [start]──→[route]   │   avg: 87ms  ✓ 100%        │
│             ↓         │   ─────────────────        │
│      [reflection]     │   memory-indexer   24ms ██  │
│         ↓             │   task-extractor  16ms █    │
│      [report]         │   pattern-finder   16ms █  │
│                       │   report-writer   15ms █    │
│   节点实时高亮          │                              │
│   执行完→绿色           │   [trace 数据实时更新]         │
│   执行中→黄色           │                              │
│   失败→红色             │                              │
└───────────────────────┴─────────────────────────────┘
```

### 核心功能

| 功能 | 说明 |
|------|------|
| **Workflow Editor** | LLM 生成 → 可视化 DAG，用户可拖拽调整 |
| **实时执行** | 节点逐个高亮执行，进程启动/完成都有动画 |
| **Trace 仪表盘** | 每个节点的 avg_ms、成功率、调用次数 |
| **历史回放** | 点任意一次执行，看当时的 DAG 状态 |

### 数据链路

```
COGTOME 执行
    ↓
trace-logger 写 JSONL
    ↓
WebUI 读 trace → 渲染 DAG 高亮 + 仪表盘
```

## 九、结论

COGTOME 的价值不在于"自己优化自己"，而在于：

> **给 LLM 提供一个带仪器仪表的执行环境，让它能精确地设计、执行、回溯和优化工作流。**

用户（或 LLM）负责规划，COGTOME 负责执行和观测，分工清晰。
