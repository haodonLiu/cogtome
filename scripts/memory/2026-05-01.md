# 2026-05-01

## trace-logger Integration — COGTOME Engine

### What was done

Integrated `trace-logger` into the COGTOME engine's `execute()` function so every motif execution is automatically traced.

### Changes to `src/engine/mod.rs`

1. **Added imports** (top of file):
   ```rust
   use std::io::Read;
   ```

2. **Modified `emit_trace` method** (replacing the old blocking file-I/O version):
   - Now spawns a background `std::thread` that calls `trace-logger`
   - Passes stdin JSON with all required fields: `trace_id`, `skill`, `date`, `started_at`, `completed_at`, `duration_ms`, `status`, `nodes`
   - Uses `std::process::Stdio::piped()` for stdin/stdout
   - Drops stdin after writing to signal EOF to the bash script
   - Reads and logs stdout output (debug level)
   - Thread is joined (not detached) so traces complete before process exit

3. **Fixed wallclock time bug**:
   - The old code used `start_time.elapsed()` (duration since execution start) as epoch seconds — resulting in wrong timestamps
   - Now uses actual `SystemTime::now()` passed from the caller
   - `completed_at` is computed from a fresh `SystemTime::now()` call

### Key design decisions

- **`std::thread::spawn` over `tokio::spawn`**: `tokio::spawn` tasks are dropped when the Tokio runtime shuts down. Since `cargo run` exits quickly after execution, fire-and-forget async tasks never ran to completion. Background threads survive process shutdown and are properly joined.
- **stdin drop before stdout read**: The bash script needs stdin closed to know when JSON input is complete. The stdin handle is dropped inside an explicit block before stdout is read.
- **Non-blocking for main execution**: The thread runs in the background; `emit_trace` returns immediately. The main async flow is unaffected.
